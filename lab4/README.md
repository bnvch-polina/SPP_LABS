# SPP Lab 4 - Dynamic Thread Pool (test automation)

Лабораторная работа 3 посвящена разработке собственного пула потоков, который динамически масштабируется в зависимости от нагрузки (очереди тестов).

## Структура решения

- `TestingFramework` - библиотека атрибутов, assert-утилит и инфраструктуры тестирования.
- `ProjectForTesting` - тестируемый проект (Todo-сервис и связанные сущности).
- `TestsForProject` - набор тестов для проекта.
- `TestRunner/DynamicThreadPool.cs` - модуль собственного динамического пула потоков.
- `TestRunner/TestExecutor.cs` - запуск тестов, интеграция с динамическим пулом, мониторинг.
- `TestRunner/TestRunnerMain.cs` - CLI-программа запуска и моделирования нагрузки.

## Что реализовано по требованиям

- Собственный `Thread Pool` без использования `ThreadPool` и `Task.Run`.
- Настраиваемые границы количества потоков: `min` и `max`.
- Автоматическое масштабирование при росте очереди и при превышении ожидания задач.
- Адаптивное сжатие: завершение простаивающих потоков.
- Неравномерная нагрузка: интервалы бездействия, пики, одиночные подачи.
- Отказоустойчивость: логирование ошибок, обнаружение и замена зависших воркеров.
- Мониторинг в реальном времени состояния пула и очереди.
- Поддержка 50+ запусков тестов для демонстрации динамического масштабирования.

## Сборка

```powershell
dotnet build "SPP_LAB_1_TRIAL.sln"
```

## Базовые режимы запуска

### 1) Последовательный запуск

```powershell
dotnet run --project "TestRunner/TestRunner.csproj" -- --mode sequential
```

### 2) Параллельный запуск (системный)

```powershell
dotnet run --project "TestRunner/TestRunner.csproj" -- --mode parallel --maxdop 8
```

### 3) Динамический пул (основной режим лабораторной)

```powershell
dotnet run --project "TestRunner/TestRunner.csproj" -- --mode dynamic --simulateload true --repetitions 3 --minthreads 2 --maxthreads 8 --idlems 900 --queuewaitms 300 --hangms 3500 --monitorms 300
```

Пример выше выполняет не менее 50 запусков тестов (для текущего набора тестов это 60 запусков).

## ЛР4: дополнительные требования (демо в запуске)
По умолчанию `TestRunner` запускается в `mode=dynamic` и применяет фильтр к тестам по метаданным:
- `--filtercategory` (по умолчанию `FilterDemo`)
- `--filterauthor` (по умолчанию `Alice`)
- `--minpriority` (по умолчанию `2`)

## Параметры динамического режима

- `--repetitions` - количество повторов полного плана тестов.
- `--minthreads` - минимальное число потоков в пуле.
- `--maxthreads` - максимальное число потоков в пуле.
- `--idlems` - время простоя потока до завершения (мс).
- `--queuewaitms` - порог ожидания задачи в очереди для масштабирования вверх (мс).
- `--hangms` - порог "зависания" воркера (мс), после которого запускается замена.
- `--monitorms` - интервал мониторинга состояния пула (мс).
- `--simulateload` - включение неравномерной модели подачи тестов (`true/false`).

## Демонстрация для отчета

Рекомендуется выполнить 3 запуска:

1. `sequential`  
2. `parallel`  
3. `dynamic` с `--simulateload true` и `--repetitions 3`  

Сравнить:

- общее время выполнения (`Elapsed`);
- динамику количества потоков в логах (`Worker #... started/retired`);
- метрики мониторинга (`queue`, `workers`, `busy`, `done`, `failed`, `recovered`).

## Примечание по fail-тестам

В наборе тестов есть демонстрационный таймаут-тест (`TimeoutDemo`), который может завершаться как `TIMEOUT` по задумке. Это нормально для демонстрации работы таймаутов и отказоустойчивости исполнителя.
